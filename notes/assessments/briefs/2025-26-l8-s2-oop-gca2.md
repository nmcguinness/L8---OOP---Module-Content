## 1. Overview

Build a **multi-user client-server application** backed by a relational database in designated groups across **four staged milestones**. The system will progress from a local DAO layer through to a networked, multithreaded Java application with a comprehensive test suite and binary file storage capabilities.

- Select a domain theme and model realistic entities with validated fields.
- Design and implement a full CRUD Data Access Layer using JDBC and the DAO pattern.
- Build a multithreaded server that handles multiple simultaneous clients using JSON as the communication protocol.
- Implement binary file upload and retrieval between clients and the database, storing binary data as BLOBs with associated metadata.
- Write a JUnit 5 test suite covering all implemented features, with ≥70% line coverage demonstrated using the IntelliJ IDEA coverage runner.
- Demonstrate and defend your work at four live stage interviews.

> Each group **must** email your lecturer a 150–200 word domain description and a list of proposed entities by **1:00 PM Wednesday of Week 5 for approval** (one email per group; include all names and student IDs). Begin significant code or database work only after written approval. Any later domain change requires re-approval by email.

## 5. Stage 1 — DAO Layer and Full CRUD

**Deadline: Sunday 8th March | Mandatory Gate — not graded**

### Overview

Stage 1 establishes the foundation of the system. You will model your domain entities, set up a relational database, and implement a complete DAO layer using the JDBC API. The service layer must depend on a DAO interface, not on the JDBC implementation class directly. All SQL must use `PreparedStatement` — string concatenation in SQL is not permitted at any stage. You must also implement bidirectional JSON conversion for each entity and produce an architecture diagram showing the intended structure of the full system.

This stage is a mandatory gate. All features listed below must be demonstrated to your lecturer before you may proceed to Stage 2. Stage 1 features must remain working at all subsequent demos.

### Required Features

| # | Feature | Specification |
| :-: | :-- | :-- |
| F1 | Entity and Database Setup | Define a DTO class for each entity with encapsulated, validated fields. Create a `mysqlSetup.sql` file that recreates the schema and populates at least 10 rows of seed data from scratch. |
| F2 | DAO Interface and JDBC Implementation | Define a `XxxDao` interface. Implement it in `JdbcXxxDao`. The service layer must depend on the interface only. `PreparedStatement` required throughout — no SQL string concatenation. |
| F3 | Get All Entities | `getAllXxx()` returns a `List<T>` of all records. |
| F4 | Get by ID | `getXxxById(int id)` returns an `Optional<T>` containing the populated DTO if found, or `Optional.empty()` if not found. Never return `null`. |
| F5 | Delete by ID | `deleteXxxById(int id)` removes the record and returns a boolean indicating success. |
| F6 | Insert Entity | `insertXxx(T entity)` inserts a new record and returns the populated DTO including the auto-generated ID from `getGeneratedKeys()`. |
| F7 | Update Entity | `updateXxx(int id, T entity)` applies field updates and returns the updated DTO. |
| F8 | Filter with Predicate | `findXxxByFilter(Predicate<T> filter)` returns a `List<T>` of matching entities using a lambda or method reference — not a raw SQL string per filter. |
| F9 | JSON Conversion | `xxxToJson(T entity)`, `xxxFromJson(String json)`, and `xxxListToJson(List<T> list)` all correct and round-trip verified. |
| — | Architecture Diagram | One-page annotated diagram showing Client → Server → DAO → Database and the JSON protocol layer. Committed to the repo before the Stage 1 deadline. Use Mermaid markdown, if appropriate. |

## 6. Stage 2 — Client-Server Integration

**Deadline: Sunday 22nd March | Graded — 50% of GCA2**

### Overview

Stage 2 moves the DAO layer onto a server and introduces networked communication between clients and server. The server must handle multiple simultaneous clients using an `ExecutorService` — each connected client is handled on a separate thread. All client-server communication uses a JSON-based protocol. You must implement a generic `ServerResponse<T>` wrapper that standardises all server replies, carrying status, message, and data fields. A protocol document must be committed to the README describing every supported request type, its payload, and its expected response.

All Stage 1 features must be working and demonstrable at this stage. The marking scheme for Stage 2 is in Section 15.

### Required Features

| # | Feature | Specification |
| :-: | :-- | :-- |
| F10 | Multithreaded Server | Server uses `ExecutorService` to handle each connected client on a separate thread. Server does not block on a single client. |
| F11 | `ServerResponse<T>` Wrapper | All server replies use `ServerResponse<T>` carrying status, message, and data. Raw types not used. |
| F12 | Display by ID and Display All | Client sends a JSON request; server calls the DAO and returns the result as `ServerResponse<T>` JSON; client parses and displays. Each team member implements this for their own entity. |
| F13 | Add Entity | Client serialises entity data to JSON and sends an insert request. Server returns the new entity (including auto-generated ID) on success, or a structured error response on failure. |
| F14 | Delete Entity | Client sends a delete request with the target ID. Server calls `deleteXxxById()` and returns a structured success or failure response. |
| F15 | Update Entity | Client sends updated field data. Server calls `updateXxx()` and returns the updated entity. |
| F16 | Error Handling and Protocol | Structured error responses returned for all failure cases — exceptions are not propagated to the client. Protocol documented in README: each request type, payload fields, and response shape. |

## 7. Stage 3 — Binary File Handling, Protocol Completion, and Testing

**Deadline: Sunday 26th April | Mandatory Gate — not graded**

### Overview

Stage 3 introduces binary file storage and retrieval between clients and the database, and a mandatory unit test suite. At least one entity table must be extended with a `BLOB` column to store binary data (e.g. image files, audio clips, or documents relevant to your domain). Clients must be able to upload a binary file to the server, which stores it in the database, and subsequently request it back — receiving the reconstructed file. Metadata (filename, content type, file size) must be stored alongside the binary data and must be independently queryable without downloading the full payload.

The JUnit 5 test suite must be passing at this stage, covering core DAO and JSON conversion behaviour. This stage is a mandatory gate — all features listed below must be demonstrated before proceeding to Stage 4. All features from Stages 1 and 2 must remain working.

### Required Features

| # | Feature | Specification |
| :-: | :-- | :-- |
| F17 | Binary Schema Extension | Extend at least one entity table with a `BLOB` column and associated metadata columns: `file_name VARCHAR`, `content_type VARCHAR`, `file_size INT`. Update `mysqlSetup.sql` to recreate this schema. The entity DTO must include a `byte[]` field for the binary data and corresponding metadata fields. |
| F18 | Binary File Upload | Client reads a binary file from disk, Base64-encodes it, and includes it in a JSON upload request alongside the metadata fields. Server decodes the payload and stores the binary data in the database using `PreparedStatement.setBytes()` or `setBinaryStream()`. Server returns a `ServerResponse<T>` confirming the stored record including its auto-generated ID. |
| F19 | Binary File Retrieval | Client sends a retrieval request specifying a record ID. Server fetches the BLOB using `getBytes()` or `getBinaryStream()`, Base64-encodes the data, and returns it in a `ServerResponse<T>`. Client decodes the payload and reconstructs the file on disk, preserving the original filename and extension. |
| F20 | File Metadata Query | Client can request metadata (filename, content type, file size) for a stored record without downloading the full binary payload. Server returns a `ServerResponse<T>` containing metadata only — the BLOB column is not fetched for this request type. |
| F21 | Disconnect / Exit | Client sends a structured `DISCONNECT` request before closing the socket. Server logs the disconnection and releases the thread cleanly. |
| F22 | Core Unit Tests | A JUnit 5 test suite with at least 3 meaningful tests per team member. Tests must be in the codebase and passing. Required categories: (1) a DAO read method (`getAll` or `getById`); (2) an `insert` with the returned auto-generated ID verified; (3) a JSON serialisation/deserialisation round-trip. Each test must have a descriptive method name (e.g. `getPlayerById_returnsEmptyOptional_whenIdDoesNotExist`). Tests must not depend on execution order; use `@BeforeEach` with known data. Coverage threshold is not required at this stage — that is assessed at Stage 4. |

### Optional: Component K — Technical Excellence (20 marks)

**This component is entirely optional.** Students who complete only the base requirements (Components E–J) can achieve 44/50 in Stage 4. This component is for students seeking to demonstrate advanced architectural understanding.

**Objective:** Design and implement a generic service abstraction layer that allows the server to handle multiple service types polymorphically.

**Requirements:**

1. **Service Abstraction (Interface or Abstract Class)**
   - Define a `Service` interface or abstract class that establishes the contract for all services
   - Must include methods for service execution, validation, and result handling
   - Design must be extensible — adding a new service type should not require modifying existing code

2. **Multiple Concrete Service Implementations**
   - Implement at least **two distinct service types**, such as:
     - `FileUploadService` and `TaskProcessingService`
     - `FileUploadService` and `ReportGenerationService`
     - `FileUploadService` and any other domain-relevant service
   - Each service must be fully functional and tested
   - Services must encapsulate their own logic and data requirements

3. **Polymorphic Service Routing**
   - Server must accept a service request, identify the service type, instantiate the appropriate concrete service, and execute it polymorphically
   - Client specifies service type in the request (e.g., `"serviceType": "FILE_UPLOAD"` or `"serviceType": "TASK_PROCESS"`)
   - Server uses polymorphism — not conditional branching — to execute the service

4. **Design Pattern Application**
   - Apply at least one advanced pattern appropriately:
     - **Strategy Pattern**: Encapsulate service algorithms as interchangeable strategies
     - **Factory Pattern**: Instantiate the correct service type based on client request
     - **Template Method**: Define service execution skeleton in abstract class with concrete steps in subclasses
     - **Command Pattern**: Encapsulate service requests as command objects
   - Pattern choice must be justified in README and explained at demo

5. **Testing and Documentation**
   - Both services must be tested in the JUnit suite
   - README must include a dedicated section explaining the service architecture, pattern choices, and design rationale
   - Architecture diagram must be updated to show the service abstraction layer

**Assessment:** See Component K rubric in Appendix A.2. Students will be assessed on: design quality, pattern application, polymorphic implementation, functionality, and ability to explain architectural decisions at the demo.

## 10. Required OOP Features

The following requirements apply across the entire codebase. Their absence directly reduces your grade and **will be probed at every stage demo**.

| Requirement | What you must do | Guidance |
| :-- | :-- | :-- |
| **Javadoc Documentation** | All classes and non-trivial methods must have Javadoc comments. Class-level Javadoc must identify primary author and any secondary authors. Trivial methods (getters, setters, `toString()`, constructors with no logic) may omit Javadoc. | See Section 11 for required format. Example: `@author Alex Kane (primary), Bill Knight (contributor)`. Method Javadoc must describe purpose, parameters, return values, and any exceptions thrown. |
| **Optional Instead of Null** | Methods that may not find a result must return `Optional<T>` — never return `null`. Apply this to DAO methods (e.g., `getById`), service layer methods, and any method where absence of a value is a valid outcome. | `Optional.of(value)` for present values; `Optional.empty()` for absent values. Client code uses `.isPresent()`, `.ifPresent()`, `.orElse()`, or `.orElseThrow()`. Eliminates `NullPointerException` risk. |
| **Design Patterns** | Apply at least two patterns from those covered in class; justify both choices in the README. | Factory (object creation), Singleton (shared DB connection), Strategy (interchangeable encoding or filtering strategies), Observer (event propagation), Adapter (integrating incompatible interfaces). |
| **Generics** | Use generic types meaningfully in at least two places; avoid raw types throughout. | Minimum: `ServerResponse<T>` for all server replies; a generic DAO interface (e.g. `Dao<T, K>`). Also: `Optional<T>` for nullable returns. |
| **Functional Interfaces and Lambdas** | Use lambda expressions or method references in at least two distinct places. | `Predicate<T>` for entity filtering (F8); `Function<T,R>` for JSON mapping; `Comparator` via lambda in stream operations. Lambdas must replace something meaningfully. |
| **Collections** | Select the most appropriate collection type for each data structure. | `List<T>` for ordered results; `Map` for lookup by ID or metadata caching. Document your choices in the README. |
| **DRY Principle** | Eliminate duplication across DAO methods, JSON converters, and request handlers. | Extract shared SQL-building, JSON-mapping, or response-wrapping logic into helper methods. |
| **Architecture Diagram** | One-page annotated diagram showing all tiers and communication paths. Submitted at Stage 1; updated if architecture changes. | Must show: Client → Server (socket); Server → DAO → Database; JSON protocol layer; binary file upload/retrieval flow. |
